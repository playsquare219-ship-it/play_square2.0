# وثيقة شرح منطق تطبيق Play Square

## 1. نظرة عامة
Play Square هو تطبيق ويب لإدارة فرق كرة القدم وتنسيق المباريات الودية، مع تركيز على تجربة عربية وبيئة لعب جزائرية.
التطبيق بني على:
- Next.js App Router
- TypeScript
- Tailwind CSS
- Firebase Firestore
- JWT Sessions عبر ملفات تعريف الارتباط `httpOnly`

## 3. نموذج البيانات (Types)
الملف `types/index.ts` يحدد الكائنات الأساسية:

- `AppUser`: بيانات المستخدم التي تحتاجها الواجهة.
- `Team`: بيانات الفريق مثل الاسم، القائد، اللاعبين، الحالة، الإحصائيات.
- `Match`: مباراة تشمل الفريقين، الملعب، التاريخ، الحالة، والنتيجة.

### أهم الحقول
- `AppUser.teamId`: معرف الفريق الذي ينتمي إليه المستخدم.
- `AppUser.isTeamCaptain`: يحدد ما إذا كان المستخدم قائد الفريق.
- `Team.players`: مصفوفة من معرفات المستخدمين.
- `Match.team1` و `Match.team2`: كائنات الفرق الكاملة.
- `Match.dateTime`: تاريخ ووقت المباراة النصي.

## 4. المصادقة والجلسة

### 4.1 `contexts/auth-context.tsx`
هذا المكون يدير حالة المصادقة في الواجهة:
- `user`: معلومات المستخدم الحالية.
- `isAuthenticated`: حالة تسجيل الدخول.
- `refreshSession()`: يستدعي `GET /api/auth/session` لاسترجاع بيانات الجلسة من الكوكي.
- `logout()`: يستدعي `POST /api/auth/logout` ويزيل الجلسة.

### 4.2 جلسة JWT
- اسم الكوكي: `playSquareToken`
- يتم تخزينه كـ `httpOnly` خطوة أمان.
- يتم التحقق منه في API عبر `verifyToken`.

### 4.3 مسارات المصادقة
- `app/api/auth/login/route.ts`: تسجيل الدخول باستخدام البريد وكلمة المرور.
  - يتحقق عبر `verifyEmailPassword`
  - يولد توكن JWT ويخزنه في الكوكي بالمتصفح.
- `app/api/auth/session/route.ts`: استرجاع حالة المستخدم من الكوكي.
- `app/api/auth/logout/route.ts`: مسح الكوكي وإنهاء الجلسة.

## 5. تخزين البيانات

### 5.1 قاعدة البيانات
التطبيق يخزن البيانات في Firebase Firestore.

### 5.2 ملفات التكوين
- `lib/server/firebase/firestore.ts`: إنشاء اتصال Firestore باستخدام `firebase-admin`.
- `lib/server/auth/jwt.ts`: توليد والتحقق من JWT.

### 5.3 المجموعات (Collections)
- `users`: بيانات المستخدمين.
- `teams`: بيانات الفرق.
- `matches`: بيانات المباريات.
- `pending_registrations`: بيانات التسجيل المؤقت (إن وجدت).

## 6. واجهة API وخط السير الخلفي

### 6.1 `lib/client/api.ts`
هذا الملف يحتوي على دوال واجهة العميل التي تتصل بـ API:
- `getAllTeams()`
- `getTeamById(teamId)`
- `createTeam(team)`
- `getMyTeams()`
- `joinTeam(teamId)`
- `getAllMatches()`
- `saveMatch(match)`

### 6.2 فرق ونظام الفريق
- `app/api/teams/route.ts`
  - GET: استرجاع قائمة الفرق أو فريق محدد.
  - GET `?mine=true`: استرجاع فرق المستخدم الحالي.
  - POST: إنشاء فريق جديد.
  - PUT/DELETE (اختياري): تحديث/حذف فريق.
- `app/api/teams/join/route.ts`
  - POST: طلب الانضمام إلى فريق، يضيف اللاعب إلى الفريق ويحدث `user.teamId`.

### 6.3 إنشاء وحفظ المباريات
- `app/api/matches/route.ts`
  - GET: استرجاع مباريات `My Matches` الخاصة بفريق المستخدم عند تسجيل الدخول.
  - POST: حفظ مباراة جديدة ببيانات `team1` و `team2` و `dateTime`.
  - PATCH: إلغاء مباراة بواسطة `matchId`.

## 7. تدفق إنشاء الفريق

### 7.1 صفحة `app/team/create/page.tsx`
- المستخدم يملأ اسم الفريق والموقع.
- عند الإرسال، يتم استدعاء `createTeam()`.
- هذا يُنشئ الفريق في Firestore ويحدّث حساب المستخدم بـ:
  - `teamId`
  - `isTeamCaptain: true`
- ثم يعيد التوجيه إلى صفحة الفريق.

### 7.2 صفحة `app/team/join/page.tsx`
- المستخدم يختار فريقاً من قائمة الفرق المتاحة.
- عند الانضمام، يتم استدعاء `joinTeam(teamId)`.
- يتم تحديث المستخدم ليصبح عضواً في الفريق.

## 8. تدفق إنشاء المباراة

### 8.1 الشروط المسبقة
- يجب أن يكون للمستخدم فريق أو يقوم بالحجز دون فريق.
- إذا لم يكن لديه فريق، تظهر له شاشة إنشاء/انضمام أولاً.

### 8.2 صفحة `app/matches/create/page.tsx`
توفر حالتين رئيسيتين:
1. **اختيار فريقك الخاص**
   - loads `myTeams` عبر `getMyTeams()`.
   - يختار المستخدم أحد فرقّه من القائمة.
2. **الحجز دون فريق**
   - ينشأ فريق مؤقت يحمل `id` يبدأ بـ `booking_`.
   - ينتقل مباشرة إلى صفحة تفاصيل المباراة.

### 8.3 أوضاع البحث عن خصم
- `instant`: اختيار خصم بسرعة من قائمة الفرق المتاحة.
- `random`: اختيار عشوائي مع تأثير بصري.
- `search`: البحث عن فريق باسم.

### 8.4 حفظ المباراة
- عند تأكيد الخصم وتعبئة التفاصيل، يُستدعى `saveMatch()`.
- `saveMatch()` يُرسل إلى `/api/matches` حزمة البيانات التالية:
  - `team1`, `team1Id`
  - `team2`, `team2Id`
  - `stadium`, `wilaya`, `baladia`
  - `dateTime`
  - `createdByUserId`

- الخادم بدوره يضمن إدراج الفريقين عبر `upsertTeam()` ثم ينشئ المباراة في Firestore.

## 9. الصفحة الرئيسية ووضعية عرض المباريات

### 9.1 صفحة `app/home/page.tsx`
تعرض:
- إحصائيات الفريق الخاص بالمستخدم.
- أزرار سريعة لإنشاء مباراة أو الذهاب إلى صفحة الفريق.
- تبويبات:
  - `All Matches`: يعرض فقط المباريات ذات الفرق الحقيقية.
  - `Upcoming`: يعرض المباريات المجدولة.
  - `My Matches`: يعرض المباريات الخاصة بالمستخدم، بما في ذلك الحجز بدون فريق.

### 9.2 `My Matches` وكيفية جلب البيانات
- تبويب `My Matches` يطلب `GET /api/matches` من الخادم.
- الخادم يقرأ `playSquareToken` من الكوكيز، ثم يتحقق منه ويرجع `user.teamId`.
- إذا كان المستخدم لديه فريق، يتم استدعاء `getMatchesByTeamId(user.teamId)` ويُعرض فقط المباريات الخاصة بهذا الفريق.
- إذا لم يكن لدى المستخدم فريق، فترجع الـ API جميع المباريات كنسخة احتياطية.
- إلغاء المباراة يتم من خلال `PATCH /api/matches` مع جسم الطلب `{ matchId, action: 'cancel' }`.

### 9.2 سبب ظهور الحجز بدون فريق فقط في `My Matches`
- يتم التعرف على الحجز بدون فريق عبر `id` يبدأ بـ `booking_`.
- تلك المباريات لا تُعرض في تبويب `All Matches`.
- تبويب `My Matches` يعرضها إذا كان المستخدم صاحب الإنشاء أو أحد الفرق.

## 10. إدارة الحالة والواجهة

### 10.1 المكونات الأساسية
- `components/ui/`: مكتبة مكونات واجهة مرنة (Button, Input, Select, Dialog, Toast...).
- `components/stats-card.tsx`: بطاقة عرض إحصائيات الفريق.
- `components/match-card.tsx`: عرض مباراة واحد.
- `components/bottom-nav.tsx`: شريط التنقل السفلي.

### 10.2 `useToast` و `useAuth`
- `useToast`: يستخدم لإظهار تنبيهات الحالة في الواجهة.
- `useAuth`: يوفر بيانات المستخدم الحالية وجلسة المصادقة.

## 11. ملخص منطق التطبيق

1. المستخدم يدخل التطبيق.
2. إذا لم يكن مسجلاً، يتم توجيهه إلى `/auth`.
3. بعد التسجيل/الدخول، يتم تحميل الجلسة عبر `/api/auth/session`.
4. إذا كان للمستخدم فريق:
   - يعرض له `Home` إحصائيات فريقه.
   - يمكنه إنشاء مباراة أو عرض فريقه.
5. يمكنه إنشاء فريق جديد أو الانضمام لفريق قائم.
6. عند البحث عن مباراة، يختار:
   - فريقه من القائمة، أو
   - حجز بدون فريق.
7. يتم اختيار خصم عبر الوضع المطلوب.
8. يتم تحديد تفاصيل المباراة وحفظها في Firestore.
9. تظل المباراة مرئية في الصفحة الرئيسية وفقاً لحالتها.

## 12. أين تُخزَّن البيانات بالضبط؟

- المستخدمون: `users`
- الفرق: `teams`
- المباريات: `matches`
- الجلسة: محفوظة في كوكي `playSquareToken`
- أي عمليات منطقية: تتم عبر API routes في `app/api/`

## 13. نقاط مهمة يجب أن تعرفها

- التطبيق لا يخزن بيانات المستخدم نهائياً في الواجهة؛ كل شيء يُحمّل من API عند الحاجة.
- واجهة `lib/client/api.ts` هي الطبقة الوحيدة التي يجب أن تتعامل معها معظم الصفحات.
- منطق مصادقة الجلسة منفصل عن منطق إنشاء الفرق والمباريات.
- الحجز بدون فريق يُمثل فريقاً مؤقتاً يتم إنشاؤه على العميل ثم حفظ المباراة.

## 14. توصيات للتطوير المستقبلي

- إضافة تحقق Firestore Rules أكثر صرامة.
- تمكين تسجيل الدخول عبر Google بالكامل إذا لم يكن موجوداً بعد.
- إضافة صفحة `matches/[id]` لعرض تفاصيل المباراة.
- تحسين فلترة `My Matches` لتستند إلى `createdByUserId` و`players` بدلاً من الأسماء فقط.
- إضافة وظيفة تحديث حالة المباراة (`completed`, `cancelled`).
